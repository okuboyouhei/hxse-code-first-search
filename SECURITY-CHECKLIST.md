## 1. 依存ライブラリの鮮度

HXSEがバンドルしている外部ライブラリは **htmx のみ**（assets/htmx.min.js）。他は自前JS（assets/hxse.js）。

| 項目 | 確認方法 | 結果 | 対応要否 |
|---|---|---|---|
| htmx最新版確認 | Web検索「htmx latest version」→ バンドル版と比較 | | |
| htmx既知脆弱性（CVE）確認 | Web検索「htmx CVE」で当年の報告を確認 | | |
| 自前JSのレビュー | assets/hxse.js の前回以降のdiffを確認 | | |

**判断基準:** パッチバージョン差は経過観察可。マイナーバージョン差またはセキュリティ修正を含む場合は更新リリースを計画する。

## 2. 公開エンドポイントの監査（HXSE最重要領域）

HXSEは**未ログイン公開の検索エンドポイント**を持つ（`/wp-json/hxse/v1/search`、`permission_callback => __return_true` は公開検索として意図通り）。認証がない分、入力検証がすべて。

- [ ] RESTエンドポイントの全受信パラメータ（keyword / facet / page / display / schema_id 等）にサニタイズがあるか
- [ ] `schema_id` の検証: 存在しないID・想定外の値でエラーや情報漏洩が起きないか
- [ ] WP_Queryに渡る値の型・ホワイトリスト検証（orderby / post_type などが利用者入力から直接組まれていないか）
- [ ] レスポンスに非公開情報（下書き・非公開投稿・カスタムフィールドの意図しない値）が混ざらないか
- [ ] 大量リクエスト・巨大パラメータへの耐性（posts_per_page の上限、キーワード長）

## 3. 外部フェッチ（api / rss / xml ソース）

- [ ] すべての `wp_remote_get` / `wp_remote_post` に `'reject_unsafe_urls' => true` があるか（**v1.9.1で導入。第1回シリーズ横展開の恒久チェック**）。新規に外部リクエストを追加した箇所も同様
- [ ] `hxse_do_remote_fetch()` のトークンがGETパラメータ付与（`_token`）である点: トークンがエラーログ・キャッシュファイル・デバッグ出力に平文で残っていないか
- [ ] 外部レスポンスのパース（JSON / RSS / XML）: XXE対策（XML外部エンティティの無効化）、想定外構造でのPHPエラー
- [ ] フェッチ失敗時のエラーメッセージにエンドポイントURLやトークンが露出しないか

## 4. ファイルキャッシュの監査

キャッシュは `hxse_cache_file_path()` で `sanitize_file_name()` を通す設計（パストラバーサル対策済み）。この防御が維持されているかを確認する。

- [ ] キャッシュパス構築のすべての経路が `hxse_cache_file_path()` を通っているか（直接パス連結の追加がないか）
- [ ] `sanitize_file_name()` の適用が外れていないか
- [ ] キャッシュディレクトリに `.htaccess` / `index.php` 等の直接アクセス防止があるか（キャッシュJSONに外部APIの応答が入るため）
- [ ] キャッシュ内容に認証情報・トークンが保存されていないか

## 5. 出力エスケープ・テンプレート

- [ ] `templates/` 配下の全 `echo` に `esc_html()` / `esc_attr()` / `esc_url()` が適用されているか（検索キーワードの再表示は反射型XSSの典型経路）
- [ ] 検索結果のハイライト処理（ある場合）がHTMLを壊さずエスケープしているか
- [ ] custom テンプレート機構でユーザー定義テンプレートに渡る値の安全性

## 6. nonce / 権限チェック

- [ ] 管理画面側の設定・操作系エンドポイント（ある場合）に `check_admin_referer()` + `current_user_can()` があるか
- [ ] 公開検索以外に `__return_true` の permission_callback が増えていないか
- [ ] 前回レビュー以降に追加されたエンドポイント（git logで特定）の検証

## 7. 自動ツール実行

- [ ] Plugin Check実行 → ERROR 0 / WARNING一覧を記録
- [ ] `WP_DEBUG = true` で検索一連（キーワード・ファセット・ページネーション・display切替・merged sources）を実行しnotice/warningゼロを確認
- [ ] 最新WordPressバージョンでの動作確認（readme.txtの `Tested up to` 更新要否）

## 8. 前回からの差分レビュー

```bash
# 前回レビュー時のタグとの差分を取得してAIに渡す
git diff <前回タグ>..HEAD --stat
git diff <前回タグ>..HEAD -- includes/
```

- [ ] 差分に対する重点レビュー（新規コードは全項目を適用）
